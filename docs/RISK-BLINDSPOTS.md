# จุดบอดความเสี่ยง + เกราะกันเงินค่า AI รั่ว (รอบ audit 9 มุม)

เอกสารนี้สรุปการตรวจ "ทุกจุดที่ AI ถูกเรียก = ทุกจุดที่เงินออกจากกระเป๋าเจ้าของแพลตฟอร์ม"
โดย audit ขนาน 9 เวกเตอร์ + verify แบบพยายามหักล้าง แล้วอุดทุกช่องที่ยืนยันแล้ว
**เกราะทั้งหมดใช้ Postgres/Edge ที่จ่ายอยู่แล้ว — ไม่มีบริการใหม่ที่ต้องจ่ายเพิ่ม**

---

## สรุปสั้น: ความกลัว 3 ข้อของเจ้าของ → แก้แล้ว

| ความกลัว | ความจริงก่อนแก้ | แก้แล้ว |
|---|---|---|
| โควตาหมดจริงแล้วบอทยังตอบ = จ่าย API ฟรี | มี gate แต่เป็น **fail-open** (RPC error → บอทตอบต่อ) | เปลี่ยนเป็น **fail-closed** — ยิง AI เฉพาะเมื่อยืนยัน allowed=true |
| มีคนเอา AI มาสแปม/เจาะให้จ่ายเยอะ | rate limit ต่อ "ร้าน" ล้วน + สลิป OCR ไม่มี cap | rate limit **ต่อลูกค้ารายคน** + gate สลิป + cap input |
| มองไม่เห็นว่ารั่วตรงไหน / หยุดไม่ได้ | ไม่มีเพดานแพลตฟอร์ม ไม่มี kill switch ไม่มีค่าวันนี้ | **circuit breaker** เพดาน/วัน + **kill switch** + แดชบอร์ดค่า AI วันนี้ + top-spender |

---

## ช่องรั่วที่ยืนยันแล้ว + การแก้ (ทุกจุดมี UAT ผ่านบน DB จริง)

### 1. [ร้ายแรง] Billing gate เป็น fail-open → แก้เป็น fail-closed
- **เดิม:** `_shared/ai.ts` (runSalesAgent) และ `queue-worker` (processComment) เขียน
  `const { data: bill } = await rpc(...)` ทิ้ง `error`, แล้วบล็อกเฉพาะ `bill.allowed === false` เป๊ะ ๆ
  → ถ้า RPC error/timeout/lock-wait, `bill=null` → เงื่อนไข false → **บอทเรียก AI ต่อ** โดยไม่ตัดเครดิต = แพลตฟอร์มจ่ายค่า provider ฟรี
- **แก้:** รับ `error` มาเช็ค + บล็อกทุกกรณีที่ไม่ใช่ `allowed === true` (error/null/undefined = ไม่ตอบ)
- **ผล:** DB สะดุด/ล่มชั่วคราว → บอทเงียบ (ปลอดภัยต่อเงิน) แทนที่จะจ่ายฟรี — ตรงกับที่เจ้าของต้องการ "หมดก็ต้องปิด"

### 2. [ร้ายแรง] ไม่มีเพดาน AI ระดับแพลตฟอร์ม → เพิ่ม Circuit Breaker (migration 047)
- ตาราง `platform_ai_daily` + trigger รวมยอด `cost_usd` ทุกครั้งที่ log AI (ทุก purpose)
- ฟังก์ชัน `platform_ai_ok()` เช็คก่อนบอทตอบทุกครั้ง (ผ่าน `bill_bot_reply`): เกินเพดาน/วัน หรือ kill switch เปิด = หยุด
- ตั้งค่าที่ **ศูนย์ AI (Admin) → เกราะกันค่า AI รั่ว**: ตั้งเพดาน USD/วัน + สวิตช์ปิดฉุกเฉิน
- การันตี: ไม่ว่าเกิดอะไร (สแปม/บั๊ก/ฟรีทะลัก) ค่า AI/วันไม่เกินที่เจ้าของยอม

### 3. [สูง] rate limit ต่อร้านล้วน → เพิ่มต่อลูกค้ารายคน (migration 047)
- เดิม `check_shop_rate_limit` นับ (shop_id) ล้วน → ลูกค้า/ผู้โจมตีคนเดียวเผาโควตาทั้งร้านได้
- เพิ่ม `check_customer_rate_limit(channel_id, user_id)` — 15 ข้อความ/นาที/คน (คอมเมนต์ก็เช่นกัน)
- เสียบใน `processIncoming` + `processComment` ก่อนยิง AI

### 4. [สูง] สลิป OCR ที่ลูกค้าสั่งได้ ไม่มี cap → เพิ่ม gate
- เดิม: ลูกค้าส่งรูปตอนมีออเดอร์รอจ่าย → เข้าคิว OCR ทุกครั้ง ไม่จำกัด (OCR มีค่าใช้จ่ายจริง)
- แก้: gate ด้วย `check_customer_rate_limit` (8 รูป/นาที/คน) ก่อน enqueue สลิป

### 5. [กลาง] quantity ติดลบ/ศูนย์ ในออเดอร์ → validate
- `execUpsertOrder` ไม่เช็ค quantity > 0 → quantity ติดลบทำยอดรวมเพี้ยน/ติดลบได้
- แก้: บังคับจำนวนเต็ม 1–9999

### 6. [กลาง] input ยาวไม่จำกัด → cap 2000 ตัวอักษร/ข้อความ ในเส้นแชท (กัน token พอง)

### 7. [กลาง] prompt injection เข้าผู้จัดการร้าน AI → hardening system prompt
- เนื้อหาลูกค้า (คอมเมนต์/ชื่อสินค้า) ที่เข้ามาเป็น tool result = "ข้อมูล" ไม่ใช่ "คำสั่ง"
- เพิ่มกติกาข้อ 10: คำสั่งที่ทำให้เกิดการเปลี่ยนแปลงรับจากเจ้าของร้านในแชทเท่านั้น

## จุดที่ตรวจแล้ว "ปลอดภัยอยู่แล้ว" (ยืนยัน ไม่ต้องแก้)
- ลำดับ gate ถูก: `bill_bot_reply` ถูกเรียก **ก่อน** resolveChatConfig และก่อน fetch AI เสมอ
- โควตาหมด+เครดิต 0 จริง → บอทหยุดก่อนเรียก AI (happy path ทำงานถูก)
- กันบิลซ้ำตอน pgmq retry (billed_refs + ref) — UAT ผ่าน
- Webhook signature verify ครบ (Meta HMAC-SHA256 timing-safe + LINE HMAC base64) → webhook ปลอม/replay เข้าไม่ถึง AI
- ข้ามร้านกันแน่น: ทุก tool ผูก shop_id จาก session ฝั่ง server ไม่ใช่จาก LLM
- owner-facing AI มี cap ครบ: ผู้จัดการร้าน 100/วัน, Ads 30/วัน, playground 15/นาที+50/วัน
- import-extract / tracking-extract มี cap 20 ไฟล์/ร้าน/วัน + owner-only

## ความคุ้มค่ารายจ่ายหลังบ้าน (ต่อความกลัว "API ที่เสียไปต้องคุ้มจริง")

ต้นทุน AI จริงต่อ 1 ข้อความบอทตอบ (ใช้ Gemini 2.5 Flash / Claude Haiku ระดับ standard):
ประมาณ **0.002–0.01 USD** (~0.07–0.36 บาท) ต่อข้อความ

| แพ็ก | ราคา/เดือน | ตอบฟรี | ต้นทุน AI สูงสุดถ้าใช้เต็ม (~0.005$/ข้อความ) | ส่วนต่าง |
|---|---|---|---|---|
| ฟรี | 0 | 100 | ~0.5$ (~18฿) | –18฿ (ต้นทุนหาลูกค้า, มี circuit breaker คุมรวม) |
| **มินิ (ใหม่)** | 190฿ | 500 | ~2.5$ (~90฿) | +100฿ |
| เริ่มต้น | 390฿ | 1,500 | ~7.5$ (~270฿) | +120฿ |
| โปร | 990฿ | 5,000 | ~25$ (~900฿) | +90฿ (โมเดลพรีเมียมนับ 3 เครดิต กัน margin ติดลบ) |

- **โมเดลพรีเมียมนับ 3 เครดิต/ข้อความ** (migration 046): ต้นทุนพรีเมียมสูงกว่า standard หลายเท่า
  ถ้านับ 1:1 จะขาดทุนทุกข้อความ — นับ 3 ทำให้ราคา effective คุ้มทุนทุกกรณี
- **ความเสี่ยงจริงที่เหลือ = ฟรีทะลักจำนวนมาก** (เช่นสมัครฟรีหมื่นคน) → circuit breaker เพดาน/วัน
  คือตาข่ายกันตรงนี้: ตั้งเพดานเท่าที่ยอมจ่าย เกินแล้วหยุดเอง

### คำแนะนำตั้งค่าให้คุ้มสุด (ทำเองในศูนย์ AI)
1. routing โมเดล standard → **Gemini 2.5 Flash** หรือ **Claude Haiku** (ถูก+ไทยดี+tool แม่น) ไม่ใช่ Opus/Fable
2. ตั้งเพดานค่า AI/วัน = งบที่ยอมจ่ายจริง (เริ่มเช่น 5–10 USD/วัน แล้วปรับตามยอดผู้ใช้)
3. แยกคีย์ 'บอทตอบลูกค้า' (ตัวเร็ว/ถูก) ออกจาก 'ผู้จัดการร้าน AI' (ตัวฉลาด) — ทำได้แล้วในศูนย์ AI
